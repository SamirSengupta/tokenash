title: OpenCode Integration
description: Route OpenCode traffic through Tokenash for token compression, MCP tools, and cached model access. One command to wrap, one to unwrap.
---

Use `tokenash wrap opencode` to route all OpenCode LLM traffic through the Tokenash proxy with a single command. The proxy compresses context, injects MCP tools, and routes API calls to your configured backend.

## Quick start

```bash
tokenash wrap opencode
```

This starts the Tokenash proxy, injects a `tokenash` provider into OpenCode's config, registers Tokenash MCP tools, sets up RTK context filtering, and launches OpenCode through the proxy.

When you're done:

```bash
tokenash unwrap opencode
```

## What `wrap opencode` does

| Step | What happens |
|---|---|
| Provider injection | Writes a `tokenash` provider using `@ai-sdk/openai-compatible` into `opencode.json`, pointing at `http://127.0.0.1:<port>/v1` |
| Runtime env | Sets `OPENCODE_CONFIG_CONTENT` with provider + model + MCP config so OpenCode picks up the proxy provider at launch |
| Provider compatibility | Leaves `OPENAI_BASE_URL` and `ANTHROPIC_BASE_URL` untouched so OpenCode `/connect` providers keep their own routing |
| Context tool | Injects RTK (or `lean-ctx`) instructions into `~/.config/opencode/AGENTS.md` and project `AGENTS.md` |
| MCP setup | Registers the Tokenash MCP server (`tokenash_compress`, `tokenash_retrieve`, `tokenash_stats`) |
| Serena MCP | Optionally registers Serena code graph tools (`--no-serena` to skip) |
| Backup | Snapshots `opencode.json` to `opencode.json.tokenash-backup` before making any changes |
| Launch | Starts the `opencode` binary through the proxy |

## Options

```bash
tokenash wrap opencode \
  --port 8787 \            # Proxy port (default: random available port)
  --no-rtk \               # Skip RTK context tool injection
  --no-mcp \               # Skip tokenash MCP registration
  --no-serena \            # Skip Serena code graph MCP
  --code-graph \           # Include code graph in context
  --no-proxy \             # Use existing proxy instead of starting one
  --learn \                # Enable memory and live learning
  --memory \               # Enable persistent memory
  --backend anthropic \    # Set backend: anthropic, openai, anyllm
  --anyllm-provider ... \  # AnyLLM provider selection
  --region ... \           # Provider region
  -- <opencode args>       # Arguments passed to opencode binary
```

## Provider model mapping

The `tokenash` provider exposes these models, all routed through the proxy:

| Provider model | Upstream model |
|---|---|
| `tokenash/claude-sonnet-4-6` | Claude Sonnet 4.6 (200K context, 16K output) |
| `tokenash/claude-opus-4-6` | Claude Opus 4.6 (200K context, 16K output) |
| `tokenash/claude-haiku-4-5-20251001` | Claude Haiku 4.5 (200K context, 8K output) |
| `tokenash/gpt-4o` | GPT-4o (128K context, 16K output) |
| `tokenash/gpt-4.1` | GPT-4.1 (1M context, 32K output) |

The default model is `tokenash/claude-sonnet-4-6`. Change it in `opencode.json` or via `OPENCODE_CONFIG_CONTENT`.

## Environment variables

| Variable | Description |
|---|---|
| `OPENCODE_CONFIG_CONTENT` | JSON payload with provider, model, and MCP config injected by wrap |
| `TOKENASH_PROXY_URL` | Optional metadata used by the `tokenash-opencode` plugin |
| `TOKENASH_CONTEXT_TOOL` | Set to `lean-ctx` to use lean-ctx instead of RTK |

## Persistent installs

`tokenash install` supports opencode as a target for persistent provider wiring:

```bash
tokenash install apply --preset persistent-service --providers manual --target opencode
```

This writes the Tokenash provider into `~/.config/opencode/opencode.json` and keeps the proxy running on port 8787.

Provider scope is also supported:

```bash
tokenash install apply --preset persistent-service --scope provider --providers manual --target opencode
```

## How it works under the hood

1. **Config injection** — The wrap command writes a `provider.tokenash` block into `opencode.json`. The provider uses the `@ai-sdk/openai-compatible` npm package, which OpenCode already supports natively. Model mappings route requests through `http://127.0.0.1:<port>/v1`.

2. **Runtime config** — `OPENCODE_CONFIG_CONTENT` is set as an env var containing the full provider + model + MCP JSON. OpenCode reads this at startup and merges it with the on-disk config.

3. **MCP tools** — Three Tokenash MCP tools are registered: `tokenash_compress` (compress context), `tokenash_retrieve` (fetch original from CCR store), `tokenash_stats` (compression statistics).

4. **Unwrap** — Restores `opencode.json` from the pre-wrap backup. If no backup exists, strips Tokenash marker blocks from the config. Also unregisters Tokenash MCP servers.

## Optional OpenCode plugin

The `tokenash-opencode` npm package also exports an OpenCode plugin. The plugin
is additive: it exposes the `tokenash_retrieve` tool and Tokenash metadata such
as `TOKENASH_PROXY_URL`, but it does not route provider traffic by itself.

Provider traffic is still routed by the injected `tokenash` provider in
`OPENCODE_CONFIG_CONTENT` or `opencode.json`. This keeps providers configured
through OpenCode `/connect` in charge of their own credentials and base URLs.

## Troubleshooting

**OpenCode doesn't use the tokenash provider.**
Check that `OPENCODE_CONFIG_CONTENT` is set and contains the correct provider block. The wrap command prints the env vars it sets.

**Provider not found after unwrap.**
If unwrap left the provider configured, run `tokenash unwrap opencode` again, or manually restore from `~/.config/opencode/opencode.json.tokenash-backup`.

**Proxy port conflict.**
Use `--port` to select a specific port, or let the proxy auto-select an available one.
